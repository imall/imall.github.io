用 rebase 來合併分支

合併分支除了使用 merge 指令之外，用 rebase 指令也可以達到另一種形式的合併分支
為什麼會說叫做另外一種形式呢？在解答之前，直接先來看一下 rebase 指令合併分支的使用
假設目前的線圖長這樣，並且希望把 dev 分支的內容合併到 master 中

那我們可以先切換到 dev 分支後，執行 rebase 指令，完成之後，線圖應該就會成功被合併：

$ git rebase master
Successfully rebased and updated refs/heads/dev.

從結果可以看得出，整個合併的形式好像跟 merge 不一樣

什麼是 rebase？

就字面上的意思來說，rebase 就是「重新定義基底」的意思。
例如最一開始的線圖中，我們可以說 123456 這個 commit 是 dev 的基底，同時也是 master 的基底

而上面的示範，則是使用 rebase 指令 來重新定義 dev 分支的基底，把 dev 分支的基底定義為 master 分支。

如果反過來操作會怎麼樣？

到這可能會有人疑惑，明明當初 merge 指令 是說要切到 master 之後下指令去合併 dev 分支
怎麼這邊 rebase 指令 變成用 dev 去 rebase master？

先回答標題的問題，如果在 master 分支 rebase dev 的話，「最終」得到的 commit 都會是一樣的，差別在於線圖會變成這樣的形式：

$ git rebase dev   
Successfully rebased and updated refs/heads/master.


這邊可以注意到，原本 master 的基底是 123456 這個 commit，但因為被 rebase(重新定義基底) 了，這導致 master 所在的 commit 以及 master 與 123456 之間的所有 commit ，都被重新定義到 dev 身上了

所以到底是要誰 rebase 誰？

這個問題的答案取決於使用者們希望最後的線圖長成怎樣，以及每個 commit 要被放置的順序
就上面的例子來說，我希望每一個 commit 按照時間來排序的話，那我就會用第一個方式，把 dev 分支 rebase master 分支
但如果我不管 commit 順序，只希望 master 最後就是要跑在最前面，那操作方式就會反過來

但如果我全都要的話呢？

那這種合併反而適合使用 merge 指令 來操作了！
