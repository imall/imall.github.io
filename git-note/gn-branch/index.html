# 為什麼要使用分支？

在版控的過程，一直用一條主線來控制版本其實沒什麼太大的問題
不過在專案進行的過程中，多少會有一些「可能會造成主線內容難以復原」的操作需要執行
使用 Git 的分支功能就很適合來處理這樣的狀況

分支就像 《火影忍者》ナルト的影分身之術，即便分身戰鬥失敗死亡了，本體也不會受到影響。

在實際團隊合作上，就很常會使用分支來合作
像是開發新功能、改 bug、實驗不同的寫法這類可能會造成原本的資料壞掉的操作
透過分支處理這些問題，無論檔案操作得怎麼樣，原本的分支還是會維持原樣，不會受到影響
甚至可以帶入 Git Flow 註一 的概念，讓團隊操控分支的想法一致，讓開發過程更順利

即使只有一個人在進行版控，也很推薦使用 Git 的分支來處理需求

註一：
在 Git 的世界中，分支本身並不會有特殊的意義，每個人對於使用分支的方式不一定會相同。
使得 2010 年，Vincent Driessen 提出 Git Flow 的概念來定義不同的分支名稱，賦予這些分支特別的意義，讓團隊能遵循這個統一規則進行開發

# 使用分支

在 Git 要使用分支的語法很單純，使用 git branch 即可

# 查詢分支

git branch 可以讓我們查詢專案有哪些分支，並且透過 * 來提示目前所在的分支位置

我們一開始執行 git init 的時候，Git 就會幫我們建立初始分支 master ，讓我們在這個分支操作
所以我們執行 git init 後，操作 git branch 指令 ，就會看到 master 分支

上面的例子就是這樣的狀況：專案裡面只有 master 這個分支，並且目前分支位置在在 master 上

# 建立分支

如果要建立一個新的分支也很簡單，只需要在打完 git branch 指令後，再加上分支名稱即可

建立完之後，我們再執行一次 git branch ，可以看到多了一個「new-branch」的分支，而我們目前的分支位置，依然還是在 master 。

我們停留在 master 分支去建立新的分支時，就像是把原本 master 分支的內容複製一份一樣
新的分支產生了一個完全獨立的工作區域
在這裡進行任何操作，如果不合併回去 master 分支，則不會影響到 master 分支本身的內容

這裡特別標記 就像 ，是因為實際上建立分支這個行為，並不會真的複製一份資料出來，只是看起來很像而已。
分支只是 Git 的機制在判別檔案內容，就算我們建立幾十個幾百個分支，檔案內容也不會被複製那麼多份去占用到硬碟空間
而分支的概念，預計文章後面會再說明。

# 切換分支

我們如果要切換到某個分支的話，可以用 git checkout 指令來操作
執行了之後，系統會提示已經切換到的分支名稱

我們再執行一次 git branch 指令，會發現 * 跑到 new-branch 去了

我們建立一個分支之後，下一步通常就是要切過去，但要下兩個指令實在有點麻煩
實務上我常常會用下面這個指令，一次進行分支的建立與切換：

透過「切換分支」的指令，並且多一個 -b 的參數
就可以跟 Git 說：「請幫我建立一個新分支，接著直接切換過去」

# 工作目錄的檔案還沒 commit 就切換分支會怎樣？

Git 切換分支的過程，會用 checkout 過去的那個 分支內容，來 更新 暫存區 以及工作目錄
但是 checkout 前的 修改 並不會有任何影響。

舉個例子來說，假設我們有兩個分支，main 跟 feature 好了
分支都有 test.html 檔案，而原始狀態下檔案內容完全相同

如果我們在 feature 分支修改 test.html 檔案內容 (新增或刪除文字) 後，再切換到 main 分支
這些更動會原封不動的保留在 main 分支的 test.html 檔案內！

甚至在 feature 新增或是刪除檔案後，沒 commit 就切分支，這些修改也會同時出現在 main 分支中哦！

執行完指令之後，再執行看看 git branch 指令，就可以看到分支名稱被更動了

# 刪除分支

如果分支的任務已經完成，未來不再需要這個分支的話，可以使用 -d 參數進行分支的刪除
執行參數之後，會跳出 Deleted branch old-branch 的訊息，告訴你已經刪除了某個分支

使用 git branch 確認一下，沒意外的話就可以看到分支被刪掉囉～

有幾種狀況沒會沒辦法刪除分支：

  1. 目前工作目錄的分支，就在要刪除的那個分支
    Git 預設會讓我們都是依著某個分支進行資料的操作，這樣對於版本控管確實也比較規矩
    所以我們沒辦法直接刪除目前所在的分支，如果要刪的話，則必須切換到其他分支之後，再執行刪除指令

  2. 沒被合併的分支
    即使我們不在要刪除的分支上，如果這分支本身一枝獨秀
    Git 會提示這個分支還沒被合併，詢問是不是真的要刪除分支

提示上有說明，如果真的是要把分支刪掉的話，執行 git branch -D test 就能成功刪除這個分支了

# 合併分支

我們開分支的目的往往是要針對原本的內容進行修改
修改到一定的程度後，會在把分支內容併回去主要的分支
故事大概會是這樣發展：

  1. 我們在 master 分支新增了一個分支 feature
  2. feature 完成了一些更動，commit 一些版本
  3. 想把 feature 改的內容，併回去 master 內

我們可以進行幾個步驟完成這件事：

  1. 先把分支切回去 master
  2. 透過 git merge 分支名 指令宣告要合併 feature 分支

順利的話，會就會完成分支合併囉

# 為什麼要先切到 master 分支？

老實說，無論用在 feature 分支使用 git merge master ，或是在 master 分支使用 git merge feature
最終得到的內容，都會是一樣的，差別在於「究竟哪個分支要得到最完整的內容」？

以剛剛的故事情節，因為 feature 分支的用途，是為了開發新的功能
而我們定義 master 分支是主線分支，feature 只負責處理新功能
由於 git merge XXX 指令的意義是「我要把 XXX 分支的內容合併給我自己」
那我們就必須 在 master 分支 執行指令告訴 Git：
「我 (master) 要把 feature 分支的內容併回我身上！」

其實 Git 並不會嚴格要求哪個分支要合併哪個分支，分支的權重會在後續進行說明
大多數的情況下，都是為了依循人為定義的分支規則，來決定誰要合併誰。

想像一下上面的例子，feature 分支明明是為了開發新功能而建立的
如果在 feature 分支執行 git merge master ，意思就會是「我這個功能分支，要把主要分支併過來哦！」
這概念就會有點奇怪了對吧！

# 合併衝突

我們在 Git 分支觀念不清楚的情況去使用分支、合併分支，無論是一個人的使用或是團隊開發，都有機會遇到 合併衝突 的議題。
事實上我們進行分支合併的過程，並沒有辦法確保衝突一定不會發生，既然這是可能會發生的議題，那我們就要知道發生了應該怎麼處理。

但如果認為使用 Git 進行版控，遇到分支合併的狀況，一定會很常遇到衝突的話，那就是一個很嚴重的錯誤觀念了！
合併會發生衝突的原因只有一個：兩個分支的過往版本，都曾經修改了 同一個檔案的同一行文字 。

例如這個操作步驟：

  1. 在 master 開了 developer 分支
  2. master 修改 readme.txt 檔案的第一行並且 commit 一版
  3. developer 也修改 readme.txt 檔案的第一行，然後 commit 一版
  4. master 合併 developer 分支時，就會衝突

又或者實務上多人操作時：

  1. 兩個使用者分別透過 master 開了 developer 跟 feature 分支
  2. developer 修改 readme.txt 檔案的第一行並且 commit 一個版本
  3. feature 也修改 readme.txt 檔案的第一行 ，commit 一個版本
  4. master 合併 developer 分支，理應順利的合併
  5. master 再合併 feature 分支，就會發生衝突

由於修改 同一個檔案的同一行文字 ，Git 在合併的過程，就會混淆兩個 commit 都改了這行的內容，應該以哪個為主？
所以發生衝突很簡單的理由，就是協作溝通不良，導致兩個人應該在不同分支修改不同內容時，卻改到了同一個地方。

無論如何，發生問題就要想辦法去解決
我們先模擬上面第一個例子的行為進行 Git 的操作。

Git 在 merge 指令發生衝突時，會呈現出這樣的資訊，告訴我們 某個檔案 發生了衝突
最後一句話最重要，Git 提示我們 先解決衝突，然後 commit 這個已解決的結果

此時我們用 git status 指令來看一下目前的狀態，這裡可以看到第三行提示一個以前沒看過的訊息
You have unmerged paths.，代表著目前正在 尚未合併完成的狀態

解決衝突的方式有兩種

  1. 放棄合併
    直接透過上面第五行提示的指令 git merge --abort 來放棄。

可能會有人覺得，放棄很可恥，既然合併了就是要成功不是嗎？
事實上，放棄確實可恥，但卻很有用，在操作合併如果發生衝突，選擇放棄也是一個蠻正常的事情
舉例來說，你是負責 master 的人，同事負責 developer 分支
今天你在 master 把 developer 合併回來結果發生衝突，但衝突的內容因為是同事改的，你也不知道怎麼修比較好
這樣的狀況，選擇放棄合併，等同事回來再跟他討論如何修改，就會是很恰當的做法
畢竟如果改壞了，壞同事說不定還會甩鍋給你。

但如果這個同事在場，可以找到人進行討論的話，就可以透過第二種方式進行處理

  1. 手動處理合併衝突
    回頭看一下剛剛 git status 的內容的第七行的內容， Unmerged paths: 會顯示尚未合併成功的路徑檔案
    這些就是所有發生合併衝突的檔案，我們要一一的處理這些檔案的合併
    以這個例子來說，發生衝突的檔案是 readme.txt，此時我們把這個檔案打開，會看到原本的內容被 Git 進行修改

原本文件內容只有一行，發生合併衝突時，Git 會把原本的分支跟要合併的分支發生衝突的部分全部都列出來
並且會呈現很固定的格式 ，開頭會是七個 “<” 符號，加上 HEAD 文字，代表著目前的分支
結尾則會是七個 “>” 符號，加上預計合併的分支名稱，代表預計合併的分支
兩者之間會有七個 “=” 符號，進行資料的相隔。

看懂了之後，我們就可以把 <<<<<<< HEAD 、 ======= 、 >>>>>>> developer 這三行的內容刪掉，並且撰寫真正要留下來的內容
可以保留自己的版本或是保留對方的版本，甚至保留兩者的版本或是直接修改成你想要的任何內容

編輯將檔案 存檔 之後，就可以透過 add 指令把 修改完的檔案加到暫存區

當執行 git add 成功之後，執行 git status 指令，系統會提示 All conflicts fixed but you are still merging. 來告訴我們衝突已經被解決了，但是我們依然處於正在合併的狀態，同時衝突的檔案出現在我們很熟悉的 準備提交 (Changes to be committed) 的項目中。

接著，我們使用 commit 指令，commit 一個版本，並且對這個合併的行為進行一個命名
命名的內容隨便我們寫，這就是一個標準的 commit 提交版本的指令

到了這邊，我們手動解決衝突的步驟，就正式完成！

總結一下分支合併如果發生衝突的解決方式：

  1. 直接放棄

使用 git merge --abort 指令來放棄合併

  1. 手動解決衝突

1. 開啟發生衝突的檔案
2. 將系統自動增加的 <<<<<<< HEAD 、 ======= 、 >>>>>>> developer 內容刪掉
3. 修改衝突內容
4. 存檔
5. 執行 git add 指令將衝突檔案加到暫存區
6. 執行 git commit 提交合併的內容

使用 vscode 的時候，合併衝突的內容會有醒目的顏色與按鈕讓我們操作
這本質上與上述內容一樣，只是用一些顏色來區別分支內容

其中 接受目前更變 、 接受來源更變 、 接受兩者更變 這三個文字按鈕
分別對應的就是目前 (HEAD) 這個分支的內容、 要拿來合併分支 的內容，以及 兩個分支的內容
點擊之後會自動刪除系統增加的那幾行文字，並依照操作保留資料，使我們更快的去操作衝突合併

# Git 分支是什麼？是像樹枝那樣嗎？

第一次聽到分支這個詞彙時，腦袋的第一個想法是聯想到「樹枝」分岔持續長大的畫面
特別是在看到 Git 的樹狀圖之後，讓我更深信的 Git 分支就是這個意思
直到最近才知道，原來不該這樣去思考 Git 的分支

首先，我們在操作分支的過程，會新增分支使樹狀圖分岔、分支 commit 使其長大
透過圖形介面或是 git log --oneline --graph 指令會看到像是樹狀圖的東西

在版控的過程，最重要的是這個樹狀圖，而不是分支本身！

整個 Git 的結構圖，無論成長的樣子是 一根直直的，還是 分岔出很多支線
在每個線段上應該都會有很多的 commit ，這些 commit 都會有 SAH 編碼作為版本號碼
而分支所扮演的腳色，除了操作到最後長出樹狀圖之外，更重要的是針對需要的 commit 做一個指標。

就像是定義一個有名字的手指頭，去指著某個 commit 的位置。

記得一開始的建立分支指令嗎？
我們在 master 分支執行 git branch new-branch 建立了 new-branch 分支，實際上這個指令的全貌是 git branch new-branch master
用上述的觀念來說明，就會是「幫我建立一個 new-branch 指標，並且指著 master 所在的那個 commit 上」，
而一開始的指令省略第二個參數，Git 會針對 HEAD 的位置來建立分支。

既然說分支是指標的概念，代表著無論指向哪一個 commit ，都還是可以透過指令重新宣告這個手指頭去指向別的 commit

例如說，我們把主要分支合併了功能分支後，功能分支 (假如名叫 feature) 已經完成他的任務，準備分配他新的任務的話
可以使用 git branch -f feature HEAD 指令，告訴 Git 我們現在強迫把 feature 分支指向 HEAD 這個位置的 commit

-f 是 --force 的意思
如果單純執行 git branch feature HEAD ，Git 會覺得是要在 HEAD 分支建立 feature 分支，跳出錯誤表示 feature 分支已經存在囉！

# 補充

# 關於分支的觀念

初學者 (包含以前的我) 可能會認為 master 分支很偉大，不能隨便動他
要開發功能就要開新的分支才能處理，而且 master 是不能隨便刪的。不然電腦可能會爆炸！

實際上，每個分支對 Git 來說並沒有任何 意義，沒有 master 分支就比較重要，feature 分支就比較不重要的說法。
分支的意義 是我們這些操作 Git 的人所賦予的，我們賦予 master 是主要分支，feature 是功能分支

假如我們今天進行這樣的操作：
先建立一個 feature 分支，在什麼都不動的狀況下，把 master 改名叫 abc，接著再把 feature 分支改成 master
就表面上來看，原本的 master 已經變成了 abc ，而目前的 master 則是 feature 改名而來
不過因為 master 分支內容沒有任何改變，所以看起來就像是從 master 分支建立了 abc 分支一般

沒有任何分支是不能更名、移動、刪除的

只要我們知道自己為什麼要對分支「改名」、「刪除」、「移動」，以及在操作之後
我們所需要的檔案是不是會依照自己的想法變動？
這些動作又會影響到自己或團隊版控的哪些層面？

這些問題都考慮過，且知道自己在進行什麼操作，Git 根本不介意我們有這些舉動，畢竟這就是 Git 提供的功能阿！
