# 複製遠端儲存庫的內容到自己的電腦

Git Server 最方便的地方，大概就是可以跟任何人共享專案，或許是團隊、或許是陌生人
身為一個工程師界的菜鳥，在還不太會開發程式的情況下，有蠻多情境可以去抓別人提供在 Git Server 上面開源的程式碼來研究

先建立一個觀念，每一個 Git 遠端儲存庫都會有一個網址，我們需要透過這個網址把專案拉下來，以 GitHub 來說，這個網址會出現在 Code 按鈕內

# 地端分支、遠端分支、遠端追蹤分支

寫到這邊，先來介紹一下什麼是地端分支，什麼是遠端分支，什麼又是遠端追蹤分支

  1. 地端分支：在分支章節提到的所有操作的分支，都是地端分支，也就是在自己的資料夾專案中所建立的分支，都屬於這類。
  2. 遠端分支：是存在 Git Server 的分支，透過 push 指令推到 Git server 的那些分支們，屬於遠端分支一類。
  3. 遠端追蹤分：是一個很特別的分支，他的主要用途並不用來開發，而是拿來追蹤與參考用的，我們可以透過遠端追蹤分支，在地端確認目前遠端有多少個分支。

會需要這樣的區分，是因為在不同狀況下，遠端分支的數量，不一定完全等同於地端分支。
有時候我們在地端開發好專案，要放到 Git Server 與人共享的分支，可能只有一個到兩個擁有完整版內容的分支
其餘在開發階段的分支，未必需要都上傳到遠端。另一方面，我們從遠端 clone 專案下來，也可能只需要 clone 主要分支即可。

接著我們就來看看，在地端專案的操作，可以怎麼查找遠端追蹤分支

# 將本地分支的資料推到遠端

建立好連結後，就可以使用 push 指令把檔案推到遠端

第一次執行這個指令，要讓地端的 Git 知道這個分支要對應到遠端的哪個分支，所以除了 git push 之外，還要多寫一些參數

  • origin 參數設定遠端分支位置的時候所定義的名稱，當時寫了什麼 ，這邊就要寫什麼
  • -u 參數的完整設定指令為 --set-upstream ，指的是上游，這個參數的目的，是讓 Git 知道，未來目前的分支，要固定對應到遠端的某個分支

在團隊開發過程，有可能 partner push 一版內容到某個遠端分支，需要把他更新下來，則可以使用這個指令

事實上 git pull 隱含著兩個指令 ：

有沒有覺得很眼熟？
fetch 這個詞在介紹 git remote -v 的時候出現過，當時告訴我們會從哪個遠端拉回資料。
git fetch 這個指令，是用來把遠端儲存庫（remote repository）所有最新的歷史記錄（commits）、分支（branches）、標籤（tags）更新，但並不將這些更新合併到目前所在的工作目錄。
merge 就是在 合併分支 的章節提到的那個 merge。

知道這兩個指令之後，應該就能了解 git pull 的意思是：「把分支內容從遠端拿回來」+「把遠端拉回來的內容合併到目前分支中」！

# 刪除遠端分支

當我們的開發分支合併回主要分支之後，既然已經沒有存在的必要，那就可以刪掉他
如果刪除地端分支，用 git branch -d 分支名稱 即可，不過遠端分支並不會因為地端分支刪除而不見

必須另外下指令才能處理，有兩個指令可以完成這件事

  1. 用 --delete 參數

這個指令很清楚能從指令猜到是要刪除分支，應該沒什麼問題

  1. 用 push 指令完整的語法

而這個指令比較令人詫異的地方是，為什麼是 :分支名稱
還記得之前有提到 push 指令完整語法是 git push origin 地端分支:遠端分支 嗎？
在 : 前面不寫地端分支，其實也就代表著，要把空的分支推送到遠端分支中
既然是空的分支，言下之意就是要把遠端的分支變成空分支，也就是刪除囉！
