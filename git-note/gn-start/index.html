# 初始化專案

要進行版控，首先要先確定到底是哪包資料夾的內容要被 Git 控管？
確定好要控管的專案後，進到要版控的資料夾，開啟終端機執行下列指令進行 Git 的初始化

執行完後，資料夾的內容會多一個 .git 的檔案夾
這就是未來 Git 在進行所有版控功能的時候會使用到的東西
資料夾內有了 .git 之後，我們就可以說這個資料夾是一個儲存庫 (repository)
如果有什麼第三方的工具提示說要開啟 locol 端的 repository ，就是在說可以開啟這包資料夾

# 將檔案加到暫存區

我們在版控的過程，不一定是要把整包資料夾的所有檔案進行版控
在初始化之後，Git 也還不會知道我們要控管哪些檔案

我們可以透過 git add 檔名 來告訴 Git ，「這裡有一些檔案要放到暫存區囉，幫我保管這些檔案」
而後面加上 檔名，代表要請 Git 追蹤的是這個檔案。

有時候我們想一次性的把所有檔案都放到暫存區裡面，可以透過下列指令一次的把工作目錄沒有加到暫存區的檔案一次加進去

# 追蹤 (Tracked) 、未追蹤 (Untracked)，暫存 (Stage)、未暫存 (Unstage)

用 「把新的檔案加入暫存區」的想法去思考 add 可能會令人誤解為，「add 指令加入新的檔案到 Git 的指令」。
不過我們在版控的時候，除了新檔案之外，還會有修改檔案、刪除檔案的行為。

實際上所謂的暫存區，是 Git 的 索引 (index) 檔案，它存在於 .git 資料夾內的 index 這個檔案。
當我們執行 git add 指令時，就是把工作目錄異動的檔案 新增到 索引 的行為

再來建立一個觀念：
在操作工作目錄的過程中，檔案會有兩種狀態 狀態，分別是未追蹤的檔案 (Untracked files) 跟 追蹤的檔案 (Tracked files)。

  1. 未追蹤的檔案 (Untracked files) 的檔案，就是所有 不在索引 的檔案。
  2. 追蹤的檔案 (Tracked files) 存在索引 的檔案，這些檔案又可以區分成兩種狀態
  • 未暫存 (Unstage)：索引內的檔案內容 與 工作目錄內的檔案內容 不一致 的檔案
  • 暫存 (Stage)：索引內的檔案內容 與 工作目錄內的檔案內容 一致 的檔案

依據實際的情況，可以歸納出幾種檔案 狀態：

  • 剛執行完 git init 的所有檔案，以及剛新增的檔案，會是 未追蹤 (Untracked files) 的狀態。
  • 執行了 git add ，但是又被修改或是刪除的檔案，就是 未暫存 (Unstage) 的狀態。
  • 執行了 git add ，但沒有再度進行修改的檔案，則會是 暫存 (Stage) 的狀態。

綜合上述可以了解到： git add 的目的，是要將工作目錄中 有異動 (新 / 刪 / 修) 的檔案 新增到 索引 內。
無論檔案是 未追蹤 還是 未暫存 (Unstage) ，都可以透過 git add 將檔案更動成 暫存 (Stage) 的狀態
而 暫存 (Stage) 的狀態，也會是執行 commit 指令前，預計作為同一個版本的檔案們，應該要有的狀態。

我第一次接觸 add 指令，真的誤以為這只是「新增檔案到暫存區」這麼簡單，認真研究了才知道明白 add 指令的實質意義
突然解釋一大堆的名詞其實有點難以理解，不過讀搞懂了之後，對於未來 Git 的操作，會有極大的幫助
因為這是最基礎也是最重要的 Git 觀念。

# 提交 (commit) 一個新的版本

當檔案都 add 完之後，就可以透過這個指令跟 Git 說，「我這裡有整理好的檔案，請幫我把這堆檔案打包成一個版本。」
在執行之後，Git 自動會生成一段亂碼 (SHA-1) 註記。
而 雙引號 的內容，是我們要針對這個版本去撰寫一些文字，好讓之後回頭看版本內容的時候，可以看出這個版本做了什麼修改。
實際在版控的時候，這個訊息內容越清楚越好，否則到時候版本一多，會很難去辨別這個版本進行了什麼修改！

如果覺得開發程每次都要先 add 之後 再 commit 很麻煩，想一次搞定的話可以用下列的指令參數，把 add 的行為跟 commit 的行為一次搞定。

不過要注意的是，這個指令只會處理 追蹤的檔案 (Tracked files) ，如果是未追蹤的檔案 (Untracked files)，是不會被處理到的哦！千萬小心！

# 查看 Git 狀態

操作 Git 的過程，如果不確定目前檔案有哪些狀態的話，可以操作這個指令進行查看。

在一個穩定的狀態下執行指令，可以看到第一行顯示著 On branch main，代表著目前我們正在 main 這個分支
而最後一句，nothing to commit, working tree clean，代表目前工作目錄是乾淨的，沒有任何東西需要被 commit

如果有新增刪除修改檔案的狀況下執行指令，有哪些檔案是 未追蹤，哪些是追蹤的檔案，將會一目了然！
例如下面的訊息中

  • Changes to be committed，意思是已更動且準備提交的檔案，就是前面提到的 暫存 (Stage) 檔案。
  • Changes not staged for commit ，意思是已更動但尚未暫存的 未暫存 (Unstage) 檔案。
  • Untracked files 很直白的就是 未追蹤 的檔案。

同時，Git 也會透過一些訊息，告訴我們可以下哪些指令去處理這些檔案的操作。

例如在準備提交 (Changes to be committed) 的狀態的訊息
提示我們可以透過 git restore --staged 'file' 的指令，把暫存區的檔案還原為未暫存 (Unstage) 狀態。

而在已更變但尚未暫存 (Changes not staged for commit) 的狀態中：

  • (use “git add/rm file…” to update what will be committed)
  • (use “git restore file…” to discard changes in working directory)

分別是告訴我們，可以透過 git add/rm file... 指令把檔案加到暫存區，
或是透過 git restore file... 指令來取消工作目錄的更變。

# 查看版控的版本

當想查看儲存庫的版本時，可以透過 log 指令查看

不過因為介面大小有限，會希望 log 出來的資訊可以一行解決，可以用 --oneline 參數 來設定

人類的慾望是無窮無盡的，除了希望一行解決之外，還希望能看到簡單的樹狀圖
可以再加上 --graph 參數

小提示： git log 指令如果有太多 commit 需要呈現，終端機結尾會出現 ":"
如果要繼續顯示的話，可以按 enter 按鍵，若要取消的話，則可以按 Q 鍵來結束

# 狀況題

# 狀況 1：希望把暫存區 (已經 add) 的檔案還原成未暫存 (或說 add 之前) 的狀態

有時候我們放在暫存區的檔案，可能因為某些原因想取消暫存，有兩個指令可以完成這個任務。

  1. 使用 git restore --staged 檔名
  1. 使用 git reset 檔名

# 狀況 2：我修改了一堆檔案，結果改亂了，能不能復原到最原始的狀態？

可以，使用 checkout 指令來處理：
雖然 checkout 指令一般用來 切換分支，
不過如果是 checkout 檔名時，則會把 目前這個分支 commit 的檔案，直接取代掉工作目錄的檔案。

有時候開發的過程一字改了好幾個檔案，想要一次復原所有修改的檔案的話，則有兩個指令可以達成這件事情

  1. 一樣使用 git checkout
  1. 使用 git reset 的 --hard 模式

git reset 除了剛才提到把暫存區檔案還原狀態的功能之外，他在 Git 最主要的功能，則是 reset 不同的 commit
因為內容比較複雜，預計之後會獨立一個篇章來說明這個議題。

無論是使用 checkout 還是 reset 指令，都是要操控 Git 已追蹤的檔案
如果是新增的檔案，因為還是未追蹤的檔案，Git 不知道要去控管他，使用這兩個指令，是沒有辦法將他刪掉的哦！

# 狀況 3：我能在 Git 版控狀況下刪除檔案或更改檔案名稱嗎？

初學 Git 的時候，我曾經一度疑惑這個操作會不會影響到 Git 版控過程
不過實務上我們進行檔案的操作也是很常見的問題，畢竟開發過程總是會做這些事情吧！

# 刪除檔案

如果想刪除檔案，可以直接刪了他，或是透過 Git 的指令 git rm [檔案名稱] 來刪除
兩個行為的差異我們用下面這個實驗來展示：

首先我們建立直接刪掉.txt 檔案跟 git rm 刪掉.txt 檔案
分別進行相對應的操作後，透過 git status 指令看看兩個檔案的狀態

我們可以看到，使用 Git 指令刪除的檔案，會直接被放到 暫存庫 的區塊
而刪掉工作目錄的檔案，則是被 Git 視為一種 檔案更動 行為
如果需要放到 暫存庫 ，再執行一次 add 指令即可

結論：這兩個操作對我們來說都是把檔案刪掉，只對 Git 來說，檔案的狀態不同！

# 改檔名

這個狀況跟刪除檔案一樣，可以直接更改工作目錄檔案，或是透過 Git 指令 git mv 舊檔名 新檔名 來處理
這邊一樣在修改後，查看檔案的狀態

比較不同的是，如果手動更改工作目錄的檔名，Git 會視為原本的檔案已經刪掉，並且新增加了一個檔案
而使用指令更改檔名，因為已經讓 Git 去修改，Git 會知道這個檔案是 變更檔名 的狀態

不過就算是自己手動改檔名，再透過 git add 指令把檔案狀態都加回暫存區
就我們使用檔案的結果而言，也沒什麼差別，就看個人喜好使用了

# 狀況 4：有些檔案我不想再版控了

如果有些檔案正在被 Git 進行版控 (已經 commit 了)，但實際上不需要提交的話，可以執行這個指令

這會告訴 Git 「我想要 remove 這個檔案，但不是真的要從工作目錄中刪除，而是要取消檔案的控管」
實際上也可以看得出是透過 rm 指令，再加上 --cached 參數來做處理 。

執行完這個指令之後，使用 git status ，可以看到檔案本身在 Git 變成刪除的異動，但同時又處在未追蹤的新增狀態！

# 狀況 5：修改 Commit 紀錄

有時候我們 commit 一個版本後，覺得某些檔案可以進一步修改再 commit，或是某些檔案沒 commit ，甚至已經 commit 但後悔了。
又或者單純想修改一下 commit 的訊息，這些狀況其實都很常見，畢竟難免會打錯字或改變心意。

有幾種方法可以修改 commit 的紀錄

  1. 把 .git 檔案直接刪了 ((別！！
  2. 用 git rebase 來修改歷史
  3. 先用 git rest 回到上一個 commit 之後，再重新 commit 一次
  4. 使用 --amend 參數直接更動最後一次的 commit

首先，第一種方法等於直接讓專案取消版控機制，同時會刪掉所有版控的紀錄，如果非必要，實在不建議用這個方式，畢竟這麼一來，將專案進行版控的意義就沒了。
第二種方式本身有許多細節需要討論，預計之後會在介紹 (吧)
第三種方式會在第四種方式說明完之後接續著說明
第四種則相對單純一些，會在接下來的內容介紹。

# 修改上一個版本的 commit 的訊息

修改最後一個版本的訊息，就是很單純的使用 --amend 參數之後，再加上 -m 參數即可進行訊息的更動

# 修改上一個版本的 commit 的內容

如果我們在 commit 之後，發現好像要調整某些檔案內容，這樣的話可以透過這個指令來處理

先透過 --amend 參數來宣告要修改最後一次的 commit ，在加上 --no-edit  來聲明這次的 commit 。
在執行之後，暫存區的更動將會合併到最後一次的 commit 內，同時保持原來的提交訊息不變。
這在想要修正最後一次提交的錯誤或者添加遺漏的檔案時很有用。

如果想要 新增或修改檔案到上一個 commit 的話，直接修改檔案並且 add 到暫存區後，執行指令即可。

但如果要刪除上一個版本已經 commit 的檔案，則須搭配 之前 提到的指令 git rm 檔案名稱 --cached 把檔案移除 Git 管控後
再執行 git commit --amend --no-edit 指令，才能達成目的哦～

--amend 指令與看起來只是純粹的修改最後一次的 commit 而已
不過對 Git 來說，卻是一筆全新的紀錄，可以注意到每次的 --amend 都會更新最後一次 commit 的 SHA-1 編碼。
所以即使 Git 能讓我們修改歷史紀錄，也盡量不要去修改已經 push 過後的內容，免得造成團隊的麻煩哦！

# 補充

# HEAD 是什麼東西？

我們在執行 git log 指令的時候，會看到 commit 的後面 有一個 (HEAD -> main)
這個 HEAD 可以解釋成「目前所在的分支」，一般來說，HEAD 這傢伙會指向的 commit 都會有一個分支
但如果 HEAD 指向的 commit 上面沒有任何分支的話，這就是 斷頭 (Detached HEAD) 的狀況
這個狀況通常是暫時的，像是我們如果想查看某個 commit 的內容，而切換過去的話，就是一種斷頭狀態

# 什麼時候要 commit ？

這其實是個大哉問，要知道 Git 其實只是提供版控的功能
而我們什麼時候要把檔案歸類成一個版本，完全取決於我們自己

有人 commit 的很隨意，覺得把所有功能開發完之後 commit 一版
有人喜歡整理得更仔細，每一個小功能開發完成之後，都 commit 一版

這其實都沒有什麼問題
可能會影響這個問題答案的，大概是自己的個人喜好，或是跟自己一起合作的人
一個人的版控，想怎麼控都無所謂
但團隊的版控，或許就需要遵循一個規則，讓大家看到彼此的版本內容，都可以很快的理解這個版本做的事情

前陣子跟公司同事們聊到這個問題，就有提到他以前處理 bug 的時候，習慣把一堆 bug 都處理完之後一次 commit 一版
結果被提醒這樣的版本內容很難 code review，於是後來就改成每完成一個小主題，就 commit 一些檔案
這樣的方式，也能讓自己或是團隊夥伴能迅速的理解每個版本的主題內容

同事特別提到，無論是想以什麼方式進行 commit ，重要的是每一個 commit 都要能確保整包專案能順利執行！

# commit 指令，究竟會 commit 了什麼？

以前在操作 Git 的時候，常常無腦的 add 後 commit，也沒去理解這動作在 Git 中到底發生什麼事情。
雖然這個樣的操作順序已經很直白的讓我們猜到，commit 指令就是要 commit add 過的檔案內容阿！
恩… 對… 就是這麼單純 XD

我們拆解一下兩個動作：

  1. add 指令是要先把要包成一個版本的資料 放到 暫存區
  2. commit 指令則是請 Git 把放在 暫存區 的檔案，整包丟到儲存庫。

所以如果沒有先執行 add 指令處理檔案就直接 commit 的話
Git 因為不知道要控管哪些檔案，而會跳出「nothing to commit」的訊息哦！

有沒有注意到，這時候 commit 的訊息，跟上面 git status 的訊息長得完全一樣
事實上 nothing to commit 的訊息就是在告訴我們，目前 工作目錄 (working tree) 沒有新東西要進行處理！

當時沒有搞懂，為什麼我明明是在學 Git ，卻要我在某個網站去安裝軟體？不是應該教我怎麼使用 GitHub 那個網站嗎？
甚至一直到 執行 git init 、 git add 、 git commit 的過程，我還是不懂這些跟 GitHub 到底有什麼關係…
我怎麼一直都在本機裡面操作資料夾阿～終端機阿～之類的東西
這不是我想像中的 Git 啊！！！

這次刻意去重新認識 Git 之後，才終於搞懂這些令我困惑的問題

現在想起來其實覺得很有趣，就把這些話記錄下來，或許會有一些剛接觸 Git 的人也跟我一樣有這種混淆
