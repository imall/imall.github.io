# 將檔案加到暫存區

開始進行版控，我們可以透過這個指令來告訴 Git ，「這裡有一些檔案要放到暫存區囉，幫我保管這些檔案」
而後面加上 [檔名]，代表要請 Git 追蹤的是這個檔案。

有時候我們想一次性的把所有檔案都放到暫存區裡面，可以透過下列指令一次的把工作目錄沒有加到暫存區的檔案一次加進去

用 「把檔案加入暫存區」的想法去思考 add 可能會有一個謬誤，add 指令比較像是 「把狀態加到暫存區」的概念。

除了一開始提到 Git 檔案 三種 存放位置 的觀念之外
在操作工作目錄的過程中，無論是新增、刪除、修改檔案的動作，都會更變 檔案 的 狀態。
而 Git 在工作目錄的檔案狀態又會有三種不同的樣貌呈現：

  1. 待追蹤 (Untracked files) ：這就是標準的 Git 還管不到的檔案們，一般剛增加的檔案就會是這種狀態。
  2. 尚未暫存的更改 (Changes not staged for commit)：如果已經在暫存區的檔案被修改了，或是已經 commit 的檔案開始進行更動，會出現這個狀態。
  3. 將要提交的更改 (Changes to be committed)：檔案已經被放在暫存區中，可以準備 commit 的檔案，則會出現這種狀態

一開始提到例子都只是把新增的待追蹤檔案加到暫存區，除了這個狀況之外，如果已經在暫存區的檔案 被修改 或是 被刪除 的話，對於 Git 來說，都會是尚未暫存 (Changes not staged for commit) 的狀態。
而 git add 的用意，就是要把待追蹤、尚未暫存的這個檔案放到暫存區 ，讓這些檔案狀態變成 準備提交 (Changes to be committed) 的動作，這也是 commit 前的檔案應該有的狀態。

# 將檔案放到儲存庫

當檔案都 add 完之後，就可以透過這個指令跟 Git 說，「我這裡有整理好的檔案，請幫我把這堆檔案打包成一個版本。」
在執行之後，Git 自動會生成一段亂碼 (SHA-1) 註記。
而 雙引號 的內容，是我們要針對這個版本去撰寫一些文字，好讓之後回頭看版本內容的時候，可以看出這個版本做了什麼修改，所以訊息內容越清楚越好，否則到時候版本一多，會很難去辨別這個版本進行了什麼修改！

如果覺得開發程每次都要先 add 之後 再 commit 很麻煩，想一次搞定的話可以用下列的指令參數，把 add 的行為跟 commit 的行為一次搞定。

不過要注意的是，這個指令只會 add 到 更改的檔案 (Changes to be committed) ，如果是新增加的檔案 (Untracked files)，是不會被處理到的哦！千萬小心！

# 查看 Git 狀態

操作 Git 的過程，如果不確定目前檔案有哪些狀態的話，可以操作這個指令進行查看。

在一個穩定的狀態下執行指令，可以看到第一行顯示著 On branch main，代表著目前我們正在 main 這個分支
並且看到，這個 branch 已經有 25 個 commits
而最後一句，nothing to commit, working tree clean，代表目前工作目錄是乾淨的，沒有任何東西需要被 commit

如果有新增刪除修改檔案的狀況下執行指令，有哪些檔案是尚未追蹤，哪些是更改但尚未暫存，哪些又是已暫存，將會一目了然！
同時，Git 也會透過一些訊息，告訴我們可以下哪些指令去處理這些檔案的操作。

例如在準備提交 (Changes to be committed) 的狀態的訊息
提示我們可以透過 “git restore --staged ‘file’” 的指令，把暫存區的檔案還原為待追蹤狀態。

而在已更變但尚未暫存 (Changes not staged for commit) 的狀態中：

  • (use “git add/rm file…” to update what will be committed)
  • (use “git restore file…” to discard changes in working directory)

分別是告訴我們，可以透過 “git add/rm file…” 指令把檔案放到暫存區，
或是透過 “git restore file…” 指令來取消工作目錄的更變。

# 查看版控的版本

當想查看儲存庫的版本時，可以透過 log 指令查看

不過因為介面大小有限，會希望 log 出來的資訊可以一行解決，可以用 --oneline 參數 來設定

人類的慾望是無窮無盡的，除了希望一行解決之外，還希望能看到簡單的樹狀圖
可以再加上 --graph 參數

小提示： git log 指令如果有太多 commit 需要呈現，終端機結尾會出現 ":"
如果要繼續顯示的話，可以按 enter 按鍵，若要取消的話，則可以按 Q 鍵來結束

# 補充

# commit 指令，到底 commit 了什麼？

以前在操作 Git 的時候，常常無腦的 add 後 commit，其實這個動作就是透過 add 指令跟 Git 說哪些檔案準備要放到儲存庫，再使用 commit 指令請 Git 把這些檔案，整包丟到儲存庫。
如果沒有先執行 add 指令處理檔案，就直接 commit 的話，Git 因為不知道要控管哪些檔案，而會跳出「nothing to commit」的訊息哦！

有沒有注意到，這時候 commit 的訊息，跟上面 git status 的訊息長得完全一樣
事實上 nothing to commit 的訊息就是在告訴我們，目前 工作目錄 (working tree) 沒有新東西要進行處理！

# 直接刪掉工作目錄的檔案，跟透過 git rm [檔案名稱] 執行的差別在哪裡啊？

這個問題我們直接來實驗看看就知道。
首先我們建立直接刪掉.js 檔案跟 git rm 刪掉.js 檔案
分別進行相對應的操作後，透過 git status 指令看看兩個檔案的狀態

我們可以看到，透過 Git 指令刪除的檔案，會直接被放到 暫存庫 的區塊
而刪掉工作目錄的檔案，則是被 Git 視為一種 檔案更動 行為，如果需要放到 暫存庫 ，則需要再執行一次 add 指令

所以其實這兩個操作對我們來說都是把檔案刪掉，只是在 Git 看來，檔案的狀態不同罷了！

# 狀況題

# 狀況一： 希望把暫存區 (已經 add) 的檔案還原成 待追蹤 (尚未 add) 的狀態

有時候我們放在暫存區的檔案，可能因為某些原因想取消暫存，或者說版本不想把這個檔案放進去，但已經執行 add 指令了
這樣的話可以用這個指令把檔案還原成待追蹤的狀態。

這題其實在 git status 的內容中有提到，如果忘記指令的話，可以直接執行 git status 來查看。

# 狀況二：有些檔案我不想再版控了

如果有些檔案正在被 Git 進行版控 (已經 commit 了)，但實際上不需要提交的話，可以執行這個指令

這會告訴 Git 「我想要 remove 掉這個檔案，但不是從工作目錄中刪除」，而是不希望這個檔案繼續被控管，實際上也可以看得出是透過 rm 指令，再加上 --cached 參數來做處理 。

# 狀況三：希望能修改上一個版本的 commit 的內容

有時候我們 commit 一個版本後，覺得某些檔案可以進一步修改再 commit，或是某些檔案沒 commit 進去，甚至已經 commit 進去但後悔了

這樣的話可以透過這個指令來處理

“–amend” 代表修正或修改的意思。這個指令用來修改最後一次 commit 的內容，
而 “–no-edit” 代表這次的 commit 不修改訊息。
執行之後，將會把暫存區的更改合併到最後一次提交中，同時保持原來的提交訊息不變。這在想要修正最後一次提交的錯誤或者添加遺漏的檔案時很有用。

如果只是想增加檔案或是修改檔案到上一個 commit 的話，直接修改檔案並且 add 到暫存區後，執行指令即可。

但如果要移除 commit 後的某一個檔案，則須搭配之前提到的 git rm 檔案名稱 --cached 指令，把檔案移除 Git 管控後，再執行 git commit --amend --no-edit 指令
