# 什麼 Git？

「阿就… 版控工具阿？！」 這可能是大部分的人聽到 Git 之後的第一個直覺反應，畢竟業界現在提到版控這個詞，要不是提 Git 就是 SVN 。

不過到底為什麼要用 Git ？我們直接把檔案 Ctrl + A 然後 包起來複製一份，不就控好了嗎？
只要把每個檔案用日期當檔名，就可以知道是哪個時間操作的，很方便吧！

這個方法很傳統也很簡單，可能隨便叫一個剛學會電腦的小學生，都知道可以這麼操作。
不過如果我希望在一堆打包的備份檔案裡面，分辨每一包程式到底差異在哪，這可能會是很大的問題。
又如果刪了某一包的備份，突然發現那份資料很重要需要復原，可能也有還原不了的風險存在。

透過 Git ，或許很多需要辨別檔案內容、操作檔案內容、更變檔案內容、了解內容版本的問題，可以很方便的解決！

# Git 基本觀念

不知道大家有沒有搭飛機的經驗，我自己是沒有，如果接下來比喻錯了請多包含 XD

想像中搭飛機的過程，整個機場會有幾個區域，一個是機場大廳，另一個是進閘門準備登機的區域，最後則是飛機本身

航空公司的工作，就是服務一組一組進閘門的客人，引領他們登機。如果在飛機起飛之前，沒有進站的客人，就沒辦法搭上這班飛機。

我們把這個概念帶到 Git 的世界中：
整個機場就像是 Git 的專案一樣，每個人都是 Git 所控管 (或是管不到) 的 檔案。
在 Git 中也是有類似這三個區域的觀念 ：「尚未暫存區」、「暫存區」、「儲存區」：

  1. 機場大廳 的人，就像尚未暫存區的檔案。Git 無法對這些檔案進行版本控制，如果刪除或修改這些檔案，Git 無法對其進行恢復。

  2. 進站後 的人，就像是在暫存區的檔案。Git 已經知道有這些檔案的存在，並且知道這些檔案預計會納入下一次的提交 (commit) 中。這個區域中的檔案被修改或刪除，都可以將其還原到之前的狀態。

  3. 已經登機 的人，就好比是在儲存區的檔案。這些檔案已經被 Git 永久地記錄成一個又一個的版本，屬於最穩定且可追溯的狀態。

  4. 飛機起飛 ，就像把檔案上傳 (push) 到遠端的儲存庫 (例如 GitHub)。

而在開發過程的操作，大多會圍繞在 暫存 / 尚未暫存區 中，去控制每一個版本的檔案 (也就是要放到儲存區的檔案) 應該長怎麼樣
在開發完成之後，則會把暫存區的檔案整批的放到 儲存區 中。

以上就是在使用 Git 版控大致的操控邏輯與架構，這屬於 Git 觀念的基礎，接下來文章的所有指令，會一直使用到這些觀念。

# 常見問題

# Git 跟 GitHub 有什麼差別，還有像是 Git Lab 、TFS 的東西，那是 Git 的別名嗎？

這也是我剛接觸 Git 搞混的觀念，因為看到字樣上都有 Git ，就覺得那應該就是一樣的東西吧！
其實 Git 是一個版本控制的 軟體 ，而 GitHub 、 GitLab 這類則是 商業網站

雖然那些已經碰過 GitHub 沒碰過 Git 的人，大概會覺得 GitHub 就是在做版控
這個說法事實上對一半，在 GitHub 執行新增檔案、對檔案 commit 訊息及送出的動作，確實是 Git 的功能沒錯
不過 GitHub 的本質是一個 Git 的雲端空間，更精闢的說法，他是一個「Git 伺服器」。

可以想像使用 Git 就像在電腦中整理文字檔，並且分門別類放在不同資料夾的過程。
而 GitHub、 GitLab、TFS 這些東西就像 Google drive、One drive 這類東西，是可以讓我們把電腦中的檔案進行上傳的雲端空間。
本機的檔案內容跟雲端空間的檔案內容，基本上不會互相干擾。即便沒有那些雲端空間，也是可以在電腦整理這些檔案。

Git 跟 GitHub 的關係也是這樣的感覺。事實上，在安裝好 Git 之後，即使沒有網路也可以在電腦中進行資料夾內容的版控
只有想要在某個雲端空間進行備份 (或是需要多人協作) 時，才會需要用到 GitHub 這類的網站。
也因為 GitHub 屬於「雲空間」，要把電腦中的檔案「推」上去，是需要有網路才能執行的。

# 不學 Git ，直接用 GitHub、GitLab 不行嗎？

雖然這些網站已經把一些基本的 Git 指令包裝成圖形化介面，在程式開發完成之後，直接把檔案「拖拉」上網站做保存，確實很方便

不過如果想要更改歷史 commit 的資料內容怎麼辦？
開發過程中，時常會因為各種原因而產生這種需求，但這並不是直接操作 GitHub 就能處理的事情

我對於 GitHub 的看法，是一種「團隊版控的最後關卡」：
只要在自己的電腦開發的檔案，還沒「推」到 Git Server*1 的內容，都可以隨意使用 Git 指令去把版本內容改成自己想要的形狀
但如果已經「推」到 Git Server 的內容，由於團隊所有成員都是能看到資料，甚至已經把內容更新到他們的電腦中了，任意再去修改將會直接的影響其他人的開發流程。

回到正題，直接操控 GitHub 本身就是直接把檔案「推」到 Git Server 的行為，基本上是沒有反悔的空間的。

或許有人會說，我可能只需要管理自己的分支，後悔了就刪了再上傳，等確定沒事之後再合併不就得了？
我不能直接說這樣不對，只會覺得這樣很浪費 Git 提供各種讓人後悔重改的功能。

事實上，直接用 GitHub 而不學習 Git 到底行不行？
我認為答案是沒有不行，或是說看使用的人需求到哪裡？
如果直接操作 GitHub 這類 Git Server ，對自己來說已經可以做到很充足的版控，也未必一定要學 Git。

註解 1： 寫 Git Server 而不是寫 GitHub ，是因為不是每個團隊使用的 Git Server 都是 GitHub
用 Git Server 來描述會比較貼切一些。
