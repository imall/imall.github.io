當時只提到如何修改最近一次的 commit 紀錄，同時挖了一個用 rebase 改歷史的坑還沒填
來把這個操作補一補吧

使用 rebase 來修改歷史，可以下這樣的指令

rebase 本身是重新定義的意思， -i 是 interactive 的意思，代表我們要使用 rebase 的互動模式
由於 rebase 是要更動基底，所以互動模式會讓你從某一個基底 開始處理這之後的更變
上面指令的 commitID 則會使用我們想修改的 commit 的前一個 ID。

假設我目前的 log 長這樣，並且執行 git rebase -i 6e72bd7 ，意思是要告訴 Git ，我現在希望從 6e72bd7 開始重新定義之後的基底。
Git 聽到後就會自動幫我挑出 890f0e6 、 9e24364 、 3d415c5 這三個 commit ，並且問我接下來要怎麼做。

# 刪除某一個 commit

先直接執行 git rebase -i 6e72bd7 指令之後，終端機會呈現這樣的資訊，提示我們正在等我們編輯或關閉檔案

所謂的檔案，則是 Git 會透過預設的 Git 編輯器跳出的這些內容，可以看到我們寫在指令的那個 commit 並沒有出現在文字之中，而是那個 commit 之後的所有 commit 都出現了。

假設我們要刪除 9e24364 這個分支，只要把這行 刪掉 之後 存檔 即可。

成功之後，就會跳出這樣的訊息

# 回復上一動

rebase 指令屬於一個比較「危險」的指令，Git 針對一些危險的指令給了一個復原的方式

這個 ORIG_HEAD 是 Git 自動幫我們紀錄的原始位置，而 --hard 參數如果不熟的話，可以參考這篇文章

# 修改歷史 commit 內容

這個需求有幾個步驟需要執行，我們一樣使用 git rebase -i 的指令來處理

  1. 執行 git rebase -i 指令
  1. 跳出這個訊息的時候，把要修改的那個 commit 前面的參數改成 edit 或是 e ，例如我想更改 fbd51f3 的紀錄，就把他的 pick 改成 e
  1. 終端機此時會跳出這些訊息，提示我們可以使用 git commit --amend 指令來修改版本

  1. 這個指令跟修改上一個版本的 commit 的訊息 以及修改上一個版本的 commit 的內容 完全一樣，基本上就是依照這個修改囉

  2. 當修改完之後，請再執行 git rebase --continue 指令 ，系統應該會自動跳出 **Successfully rebased and updated refs/heads/[你的分支].** 的訊息

# 把多個 commit 整合成一個 commit

有時候我們因為 commit 的太瑣碎，希望把一些 commit 集結起來 (Avengers Assemble！)
這需求會在執行互動模式時，把想要集結的 commit 設定成 squash，squash 這傢伙的概念，是要把新的 commit 合併到過往 commit 上頭，
假設我們的版本順序是 1、2、3，如果想把這三個版本都合併起來的話，就在編輯器將 2 、3 這兩個版本 改成 squash 或 s ，Git 就會幫我們把 2 、3 這兩個版本 併到 1 囉！

假設我們的 log 現在長這樣：

如果我希望把 版本 4 、版本 3 合併到 版本 2 的話，可以執行這些步驟

  1. 執行 git rebase -i 6e72bd7 ，代表我們希望更改 6e72bd7 之後 的所有 commit
  1. 把編輯器版本 4 跟版本 3 的 pick 都改成 s

這邊注意一下，編輯器的版本會由「舊」到「新」陳列出來
執行 squash 的時候，第一個 (最舊的) commit 不可以改成 s！
不然就沒有分支是要「被集結」了！！

  1. 存檔之後，Git 會沒完沒了的再跳出這樣的訊息，show 出這些 commit 原本的訊息，讓我們針對這次的集結做紀錄
  1. 最後就執行 git rebase --continue 就完成囉！

# 把一個 commit 拆成多個

記得之前在聊 什麼時間點適合進行 commit 的時候，有提到如果把一大堆不相干主題的修改全部 commit 在一起時，要 code review 某個 commit 會變得很不方便。
如果只是上一個版本想要拆掉重來，用 reset 就可以很方便的達成，不過如果想拆更之前的 commit，就需要用到 rebase -i 了！

前置動作，來查看一下目前的 log，我希望拆掉 commit 訊息是三個檔案的那個 commit：

  1. 起手是還是一樣，使用 git rebase -i commitID ，記得這個 ID 會是要處理的 commit 的前一個 commitID：
  1. 接著，把想要「拆」的那個 commit 改成 edit 或是 e ：
  1. 存檔之後，系統會顯示這樣的資訊，到這邊的步驟跟開頭 修改歷史 commit 內容 完全一樣。
  1. 到這邊應該會停留在「要拆」的 commit 上，接著我們就可以下 git reset HEAD^ 指令把目前這個 commit 解開
    如果不懂為什麼的話 ，可以參考 這篇文章 小結論的部分。
  1. 此時可以用 git status 確認一下，我們執行 edit 參數的 commit 裡頭的檔案，是不是都變成 未暫存 (Unstaged) 的狀態？

仔細讀一下 status 的訊息
提及說，上一個 command 已經結束了，是 edit fb8abbf commit。
下一個 commit 是 pick 437654e ，正準備要執行。
可以注意到因為我們 reset HEAD^ 的關係，現在停留的 commit 剛好就是要拆的 commit 的前一個 commit 9b3bf83。

訊息的最後，可以看到有幾個檔案是未暫存 (Unstaged) 的狀態，這些就是原本在 fb8abbf 版本已經被 commit 的檔案了！

  1. 一步一步執行 git add 跟 git commit 吧！
  1. 再來看一下 status 的狀態，會注意到基本訊息沒什麼變動，但現在處在 nothing to commit 的狀態了！
  1. 執行 git rebase --continue 來完成這次的 rebase

# 在 commit 與 commit 插入額外的 commit

這個需求要操作的動作跟 把一個 commit 拆成多個 所做的事情大概有 87% 像。

把 commit 拆成多個，會用到 reset 拆掉當下的 commit ，而現在這個需求，只要省略 reset 的動作 ，直接處理檔案之後，執行 git add 跟 git commit -m 指令即可！

記得最後要再執行 git rebase --continue 來完成 rebase！

# 修改 commit 順序與 rebase 衝突

rebase 本身就是在進行修改歷史的動作，操作過程難免會有衝突，除了等等要示範的修改 commit 順序之外，刪除某個 commit 、在 commit 與 commit 之間插入 commit 也都有可能會發生衝突。

原因很單純：在 rebase 的過程中，當已經 rebase 完成版本的檔案與 rebase 過程修改且準備 commit 的檔案出現同一行內容不一致時，由於 Git 不清楚應該保留 rebase 過程已經 commit 的檔案還是下一步版本的更改，就會發生衝突。

這個概念跟 分支合併衝突 的原因一樣，都是在操作 Git 指令時，Git 不知道我們這些操作者想要保留的到底是哪個版本，只好訊息讓我們自己解決。

衝突並不可怕，因為不知道為什麼會發生衝突才可怕

假設我們的 log 長這樣，我在某個檔案先處理 banner 區塊，再處理 header 區塊

但我希望這兩個版本對調一下，感覺比較合情合理，步驟大概會是這樣：

  1. 執行 git rebase -i 來調整
  1. 把 commit 順序調換

原本的訊息是先 commit 41b39c6 後 commit 7a569ff (越先出現的 commit 越舊)：

把它改成這樣然後存檔

  1. 系統會直接跳 衝突訊息 出來！

告訴我們 示範衝突.html 這個檔案發生了衝突，有幾種解決方法

  • 解決衝突，並且使用 git add 指令標記衝突已解決，並且 執行 git rebase --continue 指令
  • 你可以直接不要這個 commit ，執行 git rebase --skip 即可
  • 你可以放棄這個 rebase ，執行 git rebase --abort 來放棄

如果不熟 rebase 操作的過程，直接使用最後一個建議 放棄 rebase 會是非常合理的作法
因為 rebase 一個操作不當，又不知道怎麼還原的情況，很容易造成整個專案直接壞掉無法正常執行

如果使用 git rebase --skip 指令的話，則會在專案 Git 歷史中 「拔除」目前所在的這個 commit
由於會報衝突的 commit，一定是最新的版本之前的 commit，這個做法至少可以確保最新的 commit 是沒問題的
但代價是 Git 的歷史中會少了這個版本

最後就來示範一下 使用 git add 指令標記衝突已解決，並且 執行 git rebase --continue 指令 來解決衝突吧：

  1. 把發生衝突的檔案打開，會看到熟悉的合併衝突看過的符號再度出現
    從訊息看得出：目前 commit (8195886) 的檔案，跟準備要 commit 的 (7a569ff) 檔案內容發生衝突！

此時此刻，我們處理的內容，會是預計要在 7a569ff 這個版本，也就是 commit 訊息為 header區塊 這個版本的檔案內容！

  1. 修改他吧！讓檔案呈現 header 區塊的樣貌，
  1. 接著執行 git add 指令
  1. 然後執行 git rebase --continue

這一步很重要， rebase 互動模式的過程，沒有要新增任何版本的話，就不會用到 git commit -m 的指令提交 commit
而是用 git rebase --continue 讓 rebase 繼續執行！

  1. 修改 commit 訊息
    因為我們剛剛的 add 指令 與 rebase continue ，是在處理 7a569ff header 區塊 的衝突
    系統預設會把訊息帶出來，讓我們去修改，由於改變了歷史，這個訊息能寫多就盡量寫多，讓其他團隊成員知道發生什麼事情
  1. 再執行 git rebase --continue
  1. 如果有衝突 ，重複 4 、 5 、 6 、 7 步，如果沒有的話，上一步指令完成時，則會出現 rebase 成功的訊息

# 後記

rebase 修改歷史的指令憑良心說起來，真的很複雜
即使我把 rebase 可能會用到的功能都記錄下來，也不敢直接說「這是一個看起來很難，執行起來很簡單的指令」

畢竟使用 rebase 的過程，操作步驟十分繁瑣，而且訊息繁雜，一下子要在終端機執行，一下子又會跳出編輯器要編輯訊息
執行 git add 之後如果腦袋不清楚的狀況，一個不小心 git commit 下去，可能造成非預期的結果。

話說回來，Git 版控只要有關修改歷史的行為，都建議是在沒有 push 到遠端 server 之前執行
像是自己在本機開發之後，發現自己的 commit 太零散或是太集中，用 rebase 整理一下再 push ，會是一個不錯的做法

不建議因為遠端分支太亂，就在沒跟團隊討論的情況下修改歷史後 force push 回遠端分支，這很大機率會造成團隊的困擾
畢竟 Git Server 本質會團隊共用的遠端儲存庫，更改了遠端 commit ，可能使其他成員 pull 下來的資料有問題

除非跟團隊討論過決議要執行遠端的歷史異動，或是團隊就只有自己一個人，否則修改歷史的行為務必在本機端完成後，再進行 push。

總而言之，自己開發的分支在 push 到遠端 server 之前，要 commit 幾次、reset 幾次、 rebase 幾次都沒關係
而 push 到遠端的 commit ，能不要動就不要動！
