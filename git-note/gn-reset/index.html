第一次提到 reset ，是在 這篇文章 提到
當時只是拿來處理一些狀況題，卻沒有深入介紹，這篇就來說明一下 reset 比較常用的功能！

不過 reset 指令還有一些參數值得獨立出來說明，為了使關注點聚焦，拉了一個單獨的篇章來筆記。
未來要找資料也能比較清楚一點。

我們在使用 git reset 後面接 commitID 時，是為了從目前的 commit 跳到 其他 commit ，在 反悔 commit 章節所介紹的就是這個功能

在使用 reset 時切換 commit 時，Git 需要考慮 reset 前、reset 後 的檔案差異應該要如何處理

所謂的 reset 前、reset 後 分別是在說：

reset 前：目前分支 (HEAD) 所在的 commit
reset 後：要跳轉過去的 commit

reset 指令會有三種模式供我們選擇：

  1. mixed 模式：
    這是 reset 的預設模式，會讓 reset 前 的工作目錄的資料呈現在 reset 後 的工作目錄內
    在比對前後差異後，相對於 reset 後 的工作目錄檔案，如果 reset 前有新增刪除修改檔案
    會讓這些檔案呈現 待追蹤 (Untracked files) 或是 更動但尚未提交 (Changes not staged for commit)
  1. hard 模式：
    hard 模式就很純粹明瞭，直接告訴 Git 「幫我切到某個 commit 而且不保留任何修改資料」
    執行完後，就會很單純的呈現出 reset 後 的 commit 最原始的檔案內容
    工作目錄下的檔案不會被 reset 前 的內容影響到

這邊做個小結論：

  1. 我們如果後悔 commit ，可以使用 git reset HEAD^ --mixed 回到上一個版本 ( --mixed 參數可省略)
    這樣在 reset 後，就可以保留 reset 前 的檔案
  2. 如果我們想讓分支回到某個版本最原始的狀態，則可以使用 git reset HEAD^ --hard 來執行操作

上述的內容總是使用 reset 前 、reset 後 來說明，是因為 reset 指令是可以不斷「重置」到不同的 commit 上
只要是 Git 版控的歷史已經存在的 commit，都可以使用 reset 進行「跳轉」

我曾經搞混 git branch -f 跟 git reset (commitID) 指令，因為好像都是在操控分支的指向
而實際上兩個指令本身用途不同

  1. git branch -f 用來強制操控 非 HEAD 的分支指向
    由於 非 HEAD 的分支會是 commit 狀態，不用特別考慮 工作目錄 的檔案如何處理
  2. git reset (commitID) 則是用來處理 HEAD 的分支位置
    因為 HEAD 所在分支可能會有以下狀態，所以在操控分支指向時，要多考慮不同狀態下的資料何去何從。
  • 新增檔案 (尚未列入追蹤的檔案) (untracked files)
  • 新增檔案 (已經加入索引的檔案) (tracked/staged files)
  • 修改檔案 (尚未加入索引的檔案) (tracked/unstaged files)
  • 修改檔案 (已經加入索引的檔案) (tracked/staged files)
  • 刪除檔案 (尚未加入索引的檔案) (tracked/unstaged files)
  • 刪除檔案 (已經加入索引的檔案) (tracked/staged files)
