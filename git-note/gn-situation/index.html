在使用 Git 的過程，或許會遇到一些指令執行完卻希望能還原的狀況，這篇文章來記錄一下可能會用到的狀況。
:::

檔案 add 之後，希望能還原狀態

有時候我們放在暫存區的檔案，可能因為某些原因想取消暫存，有兩個指令可以完成這個任務。

  1. 使用 git restore --staged 檔名
git restore --staged 檔名
  1. 使用 git reset 檔名
git reset 檔案名稱

git reset 除了把暫存區檔案還原狀態的功能之外，他在 Git 最主要的功能，則是 reset 不同的 commit
因為內容比較複雜，預計之後會獨立一個篇章來說明這個議題。

修改了某個檔案的一堆內容，想還原檔案原始樣貌

這個問題實際上又能區分為兩種情境

  1. 尚未將檔案 add 到暫存區

若是這樣，可以直接使用 checkout 指令來還原。
雖然 checkout 指令一般用來 切換分支，不過如果是 checkout 檔名時，則會把 目前這個分支的索引所存的內容，直接取代掉工作目錄的檔案。
在使用的時候，建議加上 -- ，以避免檔名本身剛好等同於某個分支，而直接切換了分支。

git checkout -- 檔名
  1. 已經將檔案 add 到暫存區
    上文有提到，checkout 指令其實是把 索引的資料 拿來取代掉 目前工作目錄 的資料
    但如果已經使用了 add 指令，就等同於把檔案資料，存到索引裡面，如果這時使用 checkout 指令來還原檔案
    因為索引內容已經跟工作目錄內容一致，檔案本身就不會有任何改變

這時候要復原檔案內容的話，可以搭配 第一點 提到的 reset 指令先進行 add 的還原，在使用 checkout 指令復原檔案內容

git reset 檔名
git checkout -- 檔名

無論是使用 checkout 還是 reset 指令，都是要操控 Git 已追蹤的檔案 (或者說索引已經存在的檔案)
新增的檔案因為還是未追蹤的檔案，Git 不知道要去控管他，使用這兩個指令，是影響不到新增加檔案的內容的哦！

改了一堆檔案，能一次還原到這個版本的狀態嗎？

兩個指令可以達成這件事情

  1. checkout

還沒 add 所有檔案的話，可以用上面提到的指令加上.代表要把所有在索引的檔案資料取代目前工作目錄的檔案

git checkout -- .
  1. reset –hard 模式
    透過 reset –hard 模式，直接把目前所在的這個 commit 內容，強制取代掉工作目錄的所有資料
git reset --hard

關於 reset 詳細的介紹，可以參考這篇文章

能在 Git 版控狀況下刪除檔案或更改檔案名稱嗎？

初學 Git 的時候，我曾經一度疑惑這個操作會不會影響到 Git 版控過程
不過實務上我們進行檔案的操作也是很常見的問題，畢竟開發過程總是會做這些事情吧！

刪除檔案

如果想刪除檔案，可以直接刪了他，或是透過 Git 的指令 git rm [檔案名稱] 來刪除
兩個行為的差異我們用下面這個實驗來展示：

首先我們建立直接刪掉.txt檔案跟git rm 刪掉.txt檔案
分別進行相對應的操作後，透過 git status 指令看看兩個檔案的狀態

我們可以看到，使用 Git 指令刪除的檔案，會直接被放到 暫存庫 的區塊
而刪掉工作目錄的檔案，則是被 Git 視為一種 檔案更動 行為
如果需要放到 暫存庫 ，再執行一次 add 指令即可

$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        deleted:    git rm 刪掉.txt

Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        deleted:    直接刪掉.txt

結論：這兩個操作對我們來說都是把檔案刪掉，只對 Git 來說，檔案的狀態不同！

改檔名

這個狀況跟刪除檔案一樣，可以直接更改工作目錄檔案，或是透過 Git 指令 git mv 舊檔名 新檔名 來處理
這邊一樣在修改後，查看檔案的狀態

Changes to be committed:
  (use "git restore --staged <file>..." to unstage) 
        renamed:    準備用git改檔名.txt -> 用git改檔名.txt

Changes not staged for commit:
  (use "git add/rm <file>..." to update what will be committed)        
  (use "git restore <file>..." to discard changes in working directory)
        deleted:    準備自己改掉檔名.html

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        自己改的檔名.html

比較不同的是，如果手動更改工作目錄的檔名，Git 會視為原本的檔案已經刪掉，並且新增加了一個檔案
而使用指令更改檔名，因為已經讓 Git 去修改，Git 會知道這個檔案是 變更檔名 的狀態

不過就算是自己手動改檔名，再透過 git add 指令把檔案狀態都加回暫存區
就我們使用檔案的結果而言，也沒什麼差別，就看個人喜好使用了

有些檔案我不想再版控了

如果有些檔案正在被 Git 進行版控(已經commit了)，但實際上不需要提交的話，可以執行這個指令

git rm --cached 檔名

這會告訴 Git 「我想要 remove 這個檔案，但不是真的要從工作目錄中刪除，而是要取消檔案的控管」
實際上也可以看得出是透過 rm 指令，再加上 --cached 參數來做處理 。

執行完這個指令之後，使用 git status ，可以看到檔案本身在 Git 變成刪除的異動，但同時又處在未追蹤的新增狀態！

$ git status
On branch master
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        deleted:    readme.txt

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        readme.txt
