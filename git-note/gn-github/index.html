要在 GitHub 同步地端的資料，首先要在 GitHub 建立一個 repository

  1. 找到建立 repository 的按鈕
    據我所知 GitHub 好像已經改版面很多次，每次按鈕位置都不太一樣
    不過基本上都會在顯眼的位置，就找到之後點開他

  2. 針對這個 repository 命名，點選 create repository
    一般來本地資料夾叫什麼名字，這個名稱就會叫什麼名字

  3. 完成
    repository 建立成功之後，會看到類似下面這樣的畫面，有提示教學怎麼把地端的 Git 專案與這個專案連結

再來說明這一小節的重點：新增遠端儲存庫位置
記得在一開始提到 Git 跟 GitHub 差異的時候，有講到 Git 是可以在地端獨立進行版控，不一定要上傳到雲端的
上一個步驟已經先建立了遠端專案，目前內容是空的，如果要把地端 Git 同步到遠端，勢必需要讓地端知道遠端儲存庫的位置
在地端的專案執行這個指令：

  • 名稱 ，是遠端節點的意思，一般來說是 origin ，要命其他名字也可以，只要跟團隊說好要叫什麼名字即可，通常這個名字也只會有一個。
  • 遠端分支位置，通常是 [專案網址].git

# 將本地分支的資料推到遠端

建立好連結後，就可以使用 push 指令把檔案推到遠端

第一次執行這個指令，要讓地端的 Git 知道這個分支要對應到遠端的哪個分支，所以除了 git push 之外，還要多寫一些參數

  • origin 參數設定遠端分支位置的時候所定義的名稱，當時寫了什麼 ，這邊就要寫什麼
  • -u 參數的完整設定指令為 --set-upstream ，指的是上游，這個參數的目的，是讓 Git 知道，未來目前的分支，要固定對應到遠端的某個分支

# 地端分支、遠端分支、遠端追蹤分支

寫到這邊，來介紹一下什麼是地端分支，什麼是遠端分支，什麼又是遠端追蹤分支

  1. 地端分支，是我們在分支章節裡面提到分支，就是地端分支
  2. 遠端分支，是已經存在 Git Server 的分支，稱為遠端分支
  3. 遠端追蹤分支是一個很特別的分支，剛剛使用 git push 指令建立把分支推送到遠端儲存庫的時候，在地端建立了一個特別的分支
    紀錄目前的 Git 專案中，有推過哪些分支

實際上，我們不該直接 checkout 到遠端追蹤分支，因為這個類別的分支是 Git 拿來追蹤用的，而不是拿來開發用的，千萬注意！

# 把遠端分支新的內容，更新到地端

在團隊開發過程，有可能 partner push 一版內容到某個遠端分支，需要把他更新下來，則可以使用這個指令

事實上 git pull 隱含著兩個指令 ：

有沒有覺得很眼熟？
fetch 這個詞在介紹 git remote -v 的時候出現過，當時告訴我們會從哪個遠端拉回資料。
merge 就是在 合併分支 的章節提到的那個 merge

所以 git pull 的意思是：「把分支內容從遠端拿回來」+「把遠端拉回來的內容合併到目前分支中」！

# 刪除遠端分支

當我們的開發分支合併回主要分支之後，既然已經沒有存在的必要，那就可以刪掉他
如果刪除地端分支，用 git branch -d 分支名稱 即可，不過遠端分支並不會因為地端分支刪除而不見

必須另外下指令才能處理，有兩個指令可以完成這件事

  1. 用 --delete 參數
    這個指令很清楚能從指令猜到是要刪除分支，應該沒什麼問題
  1. 用 push 指令完整的語法
    而這個指令比較令人詫異的地方是，為什麼是 :分支名稱
    還記得之前有提到 push 指令完整語法是 git push origin 地端分支:遠端分支 嗎？
    在 : 前面不寫地端分支，其實也就代表著，要把空的分支推送到遠端分支中
    既然是空的分支，言下之意就是要把遠端的分支變成空分支，也就是刪除囉！
